# Star-Skill Radar（AI 技术资产情报雷达）- 产品需求文档（PRD）

## 需求描述

### 背景
- **业务问题**：AI 开发时代，高质量的 skill、MCP、agent 工作范式等技术资产是开发效率的核心依据，但信息分散在 GitHub、官方博客等渠道，大部分开发者信息闭塞，无法持续、可靠地获知最新高价值资产。
- **目标用户**：使用 Cursor / Claude Code / Codex 等 AI 编程产品的开发者（当前为单用户自用，报告中文撰写）。
- **价值主张**：以每周一期的自动化情报报告，持续捕捉权威来源的高星技术资产与工作范式，附带竞品对比、推荐理由与"在 AI 产品中如何用"的最佳使用范式定位，消除信息差。

### 功能概述
- **核心功能**：
  1. 定时（每周）自动采集：GitHub 高星项目为主 + 官方权威源（Anthropic / OpenAI / Cursor 官方文档与博客）
  2. 分级可信度验证：官方/权威机构直接准入；社区项目需 star 阈值（≥1k）+ 活跃度 / 作者背景等多信号交叉验证，报告中标注证据等级
  3. 同类竞品对比分析与推荐理由（解释高星项目的合理性）
  4. 最佳使用范式定位：说明每个资产适用于开发全周期的哪个阶段（前期需求文档 / 中期代码审查 / 后期性能测试等）
  5. 历史去重：维护已收录清单（本地索引），已报道项目默认不重复；仅在重大更新（大版本 / 重要新特性）时以"追踪更新"形式再次出现
  6. 双格式交付：Markdown 按周归档（git 管理）+ 美观、交互友好、移动端友好的 HTML 版本
- **功能边界（包含）**：
  - 主题范围：① 全部技术资产类（skills / MCP servers / rules / hooks / subagents / 提示词库）② 工作范式与方法论（harness 工程、agent loop 设计、spec-driven / vibe coding 最佳实践）
  - 运行载体：Cursor Automations（云端 Cloud Agent，cron 每周触发）
- **功能边界（不包含 / 非本期）**：
  - 工具链与框架追踪、评测基准追踪（用户未勾选）
  - 每日更新、即时推送通知（IM/邮件）、可直接安装 skill 的自动适配（后续可扩展）
  - 多用户 / 团队协作功能
- **用户场景**：每周自动生成一期报告并推送至仓库；用户在 Cursor 内、GitHub（网页/手机 App）或 GitHub Pages 的 HTML 情报站中阅读；通过 HTML 站可快速浏览历期、按主题/阶段筛选条目。

### 详细需求
- **输入/输出**：
  - 输入：GitHub 检索结果（trending、awesome 列表、skill/MCP 仓库、star 数据）、官方文档与博客页面、历史已收录清单
  - 输出：`reports/YYYY-Www.md`（每周 Markdown 报告）+ HTML 情报站（历期索引 + 单期页面）+ 更新后的已收录清单索引
- **报告规格**：每期 3-5 个重点条目深度分析（含项目介绍、证据等级、竞品对比、推荐理由、使用范式定位）+ 若干条简讯；中文撰写
- **数据要求**：已收录清单需为结构化、可被程序读写的索引（含项目标识、收录日期、当时版本/状态，用于重大更新判断）
- **边缘情况**：
  - 某来源不可访问 / 检索失败 → 报告中标注该来源本期缺失，不阻断整体流程
  - 当周无满足可信度门槛的新条目 → 生成"本期无重点条目"的简报，不硬凑
  - 重复检测：以仓库唯一标识（owner/repo）为主键判重，改名/迁移项目按新旧标识关联

## 设计决策

### 技术方案
- **架构选择**：Cursor Automations（云端 Cloud Agent）+ GitHub 仓库为存储与发布中枢。理由：零额外基础设施、复用现有 MCP（GitHub、Firecrawl）与 Agent 能力、Git 天然提供历史版本管理，GitHub Pages 免费托管 HTML 站。
- **关键组件**：
  1. 采集与验证 SOP（仓库内指令文档，供 Automation Agent 执行时遵循）
  2. 已收录清单索引（去重与更新追踪的数据基础）
  3. 报告生成规范（Markdown 模板 + 内容结构约定）
  4. HTML 情报站（静态站点，GitHub Pages 托管，移动端友好）
  5. Cursor Automation 定义（cron 每周触发 + Agent 提示词）
- **数据存储**：全部存于 git 仓库（报告、索引、站点均为文本/静态文件），无外部数据库
- **接口设计**：无对外 API；HTML 站为纯静态浏览

### 约束条件
- **性能要求**：单次运行在 Cloud Agent 合理时长内完成（目标 < 30 分钟）；HTML 站为静态页面，秒开
- **兼容性**：HTML 站需兼容手机浏览器（响应式）；Markdown 报告需在 Cursor / GitHub 渲染正常
- **安全性**：仓库可设为私有（HTML 站若走 GitHub Pages 公开托管则报告内容公开，需用户确认取舍）
- **可扩展性**：主题范围、来源清单、推送渠道（Slack 等）均设计为可配置项，便于后续扩展

### 使用边界说明（用户明确要求交付）
- **无需常开 Cursor / 电脑**：Cursor Automations 运行于 Cursor 云端（Cloud Agent），本机关机不影响定时执行
- **前提条件**：项目必须推送至 GitHub 仓库，并授权 Cursor 的 GitHub 集成；Cloud Agent 运行消耗 Cursor 账户用量额度
- **阅读渠道**：① Cursor 内直接读 Markdown ② GitHub 网页 / 手机 App ③ GitHub Pages HTML 情报站（手机浏览器友好，推荐移动端主入口）
- **远程推送**：Automations 原生支持 Slack 通知；邮件 / 微信非原生支持，如需可后续通过 GitHub Actions / webhook 扩展（本期不含）

### 风险评估
- **技术风险**：Cloud Agent 单次运行时长/能力上限可能不足以完成全部采集+分析 → 缓解：SOP 中明确优先级（先重点条目后简讯），允许降级产出
- **依赖风险**：GitHub API 限流、Firecrawl 抓取失败 → 缓解：来源失败标注机制，不阻断整体
- **准确性风险**：LLM 生成的竞品对比 / star 数据可能失实 → 缓解：SOP 要求关键数据（star 数、作者、日期）必须来自工具实时查询而非模型记忆，并附来源链接
- **公开性风险**：GitHub Pages 公开 → 需用户在实施阶段确认仓库公开/私有策略

## 验收标准

### 功能验收
- [ ] 定时采集：Automation 每周自动运行一次，产出当期报告并提交至仓库
- [ ] 可信度分级：每个重点条目标注证据等级（官方/权威 vs 社区多信号验证），附来源链接与实时数据
- [ ] 竞品对比：重点条目含同类项目对比与推荐理由
- [ ] 使用范式：每个条目明确标注适用的开发阶段（前期/中期/后期）与使用建议
- [ ] 历史去重：已收录项目不重复出现；重大更新以"追踪更新"形式呈现
- [ ] 双格式交付：Markdown 按周归档 + HTML 情报站同步更新，手机浏览器可正常浏览与筛选

### 质量标准
- [ ] 报告数据可信：抽查条目的 star 数、链接、作者信息与实际一致
- [ ] 降级健壮：模拟来源失败场景，流程不中断且报告中有标注
- [ ] HTML 站在桌面与移动端渲染正常，历期索引可导航

### 用户验收
- [ ] 用户确认首期报告的内容质量与格式符合预期
- [ ] 使用边界说明文档交付（本 PRD 已含，README 中同步）

## 执行阶段

### Phase 1: 工程脚手架与调研（架构师直接执行）
- [ ] 仓库初始化、目录结构、治理文件（AGENTS.md / todolist.md）
- [ ] 权威来源清单调研报告（docs/research/）
- **交付物**：可用的仓库骨架 + 来源调研材料

### Phase 2: 核心能力开发（子窗口并行）
- [ ] 采集与可信度验证 SOP + 报告规范 + 去重索引机制
- [ ] HTML 情报站（静态、响应式、历期索引）
- **交付物**：Agent 可执行的 SOP 体系 + 可发布的情报站

### Phase 3: 集成与联调
- [ ] GitHub 仓库创建与推送、Pages 启用（公开/私有策略确认）
- [ ] 手动触发一次完整流程，产出首期报告，验证全链路
- **交付物**：首期报告 + 上线的 HTML 站

### Phase 4: 自动化上线
- [ ] 创建 Cursor Automation（cron 每周触发），配置 Agent 提示词
- [ ] 观察首次自动运行，验收后收尾
- **交付物**：运行中的每周自动化 + 完成报告
