# 自动化架构指南（通用，非本项目专属）

> 本文档在 Star-Skill Radar 项目收尾时沉淀，记录的是可复用于任何未来自动化项目的概念与决策框架，不是本项目的运行说明（运行说明见 `docs/automation-recipe.md`）。

## 一、三个常见理解的校准

1. **"云端脑"心智模型**：账户绑定、云端按计划自动执行、无需本地开客户端、消耗账户额度——这个理解正确，且已在 Cursor 上实测验证。但不能跨产品泛化：Claude Code、Codex 是否有对等能力、交互形态、计费方式如何，需要分别查各自官方文档确认，不能直接照搬。
2. **推送范式**：Server酱之类方案的本质，是把"给你发消息"转换成"任何程序能发一次 HTTP 请求就能触发通知"。这个范式可复用到几乎所有目标渠道（如飞书自定义机器人 webhook，甚至比微信路径更直接）。
3. **GitHub（或任何持久化存储）的核心作用**：不只是"避免版本丢失"和"免费公网托管"。最本质的机制是——**每次云端自动化运行都是全新的、无记忆的会话**，GitHub 的真正作用是把这个无状态的大脑，改造成一个跨运行有长期记忆的系统（读 ledger/index 才能"记得"过去）。此外还提供：可审计的版本历史（出问题时靠 `git log` 复原真相）、下游动作的触发源（push 触发 Actions 通知，GitHub 兼任"总线"）。

## 二、自动化系统的五要素解剖

| 要素 | 作用 |
|---|---|
| 触发器 Trigger | 什么条件下无需人动手就启动（定时/事件/条件） |
| 计算/云端脑 Compute | "大脑"在哪跑、谁出钱（账户云端配额） |
| 依据/SOP Instructions | 无记忆的大脑如何知道自己该干什么（外部化的操作手册） |
| 持久化状态 State | 大脑如何"记得"过去（数据库/仓库/文件） |
| 下游动作 Actions | 产出后自动触发什么后续（通知、发布、重建站点） |

拆解任何自动化想法，先套这五格；哪一格拆不出来，往往就是这个想法尚不成熟的地方。

## 三、六维边界判断框架

| 维度 | 适合全自动 | 需要人工把关 / 慎用 |
|---|---|---|
| 可规程化程度 | 决策逻辑能穷举成 SOP | 依赖临场审美/直觉，无法提炼成规则 |
| 来源可达性 | 素材可被程序稳定获取 | 需登录墙/强反爬/只存在人脑里 |
| 正确性可验证性 | 产出可程序校验 | 质量好坏无硬指标 |
| 失败代价与可观测性 | 代价低且容易被发现 | 代价高或错了没人知道 |
| 发布动作可逆性 | 目标私密/自用 | 目标公开、有品牌影响 |
| 单次运行自包含性 | 能拆成"醒来→读取上下文→干活→睡回去" | 任务本质持续在线、随时响应（需常驻服务架构，非 cron 形状） |

六项全部打勾 → 可直接全自动；有短板（尤其"发布可逆性"“失败代价”）→ 在"生成"与"公开发布"之间插入人工确认关卡，待运行出稳定质量记录后再收窄人工介入范围。

### 应用示例

- **日报推送飞书群**：六维全部打勾，教科书级适配，且比微信方案更简单（飞书 webhook 原生直发）。
- **AI 资讯→视频→发布**：需拆段评估——"爬取+整理"同构于本项目，适合全自动；"生成视频"可规程化程度骤降、可验证性低、发布不可逆且失败代价高，建议先做到"自动生成草稿"，发布前保留人工确认，待质量记录稳定后再考虑收窄。

## 四、账户频繁更换时的可用性策略（最低成本优先）

**风险机制**：Automation 绑定在创建它的账户上（云端账户级资源，非本地登录状态）；当前"在用"账户若没配置，触发点静默落空。

**推荐方案**：把自动化配方复制到每一个轮换账户上（一次性成本，几分钟/账户），任一账户当天有额度即可独立触发。副作用红利：已有的"当天文件已存在则跳过"幂等保护，天然让多账户同时触发也不冲突、不重复。

**残留风险**：仍需"当天至少一个账户有效"；账户彻底停用需要在新账户上重新配置一次。

**更彻底但成本更高的路径**：脱离 Cursor 账户配额体系，改用 GitHub Actions + 独立模型 API Key 运行（Cursor 提供对应 SDK 支持从 CI/CD 程序化拉起 Agent）。属于值得单独立项规划的任务，非"最低成本"答案。

**未验证项**：Cursor Automations 的 "Team" 模式是否不绑定单一成员账户、是否用共享配额，尚未验证，如有 Team 版本值得自行实测。
